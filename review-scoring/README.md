# Review Scoring

Автоматизація review scoring за SOP: PDF-експорти відгуків Amazon →
Excel-книга формату еталона (Positive / Negative / Usage / Improvement /
Products) з дворівневим групуванням фраз.

## Як це працює

1. **Парсинг PDF** — відгуки ріжуться на автора, рейтинг, заголовок, дату,
   варіацію, текст. За замовчуванням аналізується весь PDF; опційно можна
   обмежити (`max_reviews` — N найновіших, `cutoff` — лише після дати).
2. **Екстракція фраз (Claude)** — з кожного відгуку витягуються дослівні
   цитати клієнтів з категоріями: positive / negative / usage /
   improvement / who_recommended. Правила з відео-SOP закодовані в промпті
   (дослівність, подвійні фрази, пропуск беззмістовних «Perfect, A+»).
   Після екстракції кожна цитата звіряється з текстом відгуку: близькі
   парафрази автоматично виправляються на реальні слова клієнта; для решти
   модель шукає точний фрагмент у відгуку (рятує «склеєні» з двох місць
   цитати), а цитати, яких у відгуку так і не знайдено, вилучаються з
   аналізу і логуються у `verbatim_warnings.json`.
3. **Групування (Claude, два рівні):**
   - *Канонічна фраза* — майже ідентичні формулювання зливаються, голоси
     сумуються («stop bleed quick» = «It stops the bleeding quickly»).
     Похвала має два яруси: звичайний (good/well + «просто works») і
     сильний (great/excellent/outstanding/amazing…) — всередині ярусу
     зливається («works good» = «works well»), між ярусами НІ
     («works well» ≠ «works great»). Різні відтінки теж НЕ зливаються
     («quickly» ≠ «instantly/immediately», «тиждень» ≠ «ніч»,
     perfectly/fast/ok — окремі рядки). Рішення судді про злиття
     додатково проходить детермінований гейт (`merge_compatible` у
     similarity.py): якщо формулювання відрізняються змістовним словом,
     фраза стає окремим рядком у тій самій групі.
   - *Група (USP)* — канонічні фрази однієї теми; назва — найчастіше
     клієнтське формулювання. Фраза з двома темами йде за домінантним
     кутом («just apply … and it stops the bleeding» → Easy to use).
   - Судді допомагає лексичний пре-фільтр: до кожної фрази додається
     shortlist схожих існуючих рядків (`[similar: …]`), щоб потрібна
     група не губилася у великій таксономії.
   - Після кожного батча нові групи, схожі на існуючі, проходять
     контрольну перевірку «нова тема чи дублікат?» (`double_check`).
4. **Консолідація** (`consolidate`) — пас по всій таксономії: один виклик
   на категорію зливає групи-дублікати, що виникли на ранніх батчах, і
   переносить фрази, що потрапили не в ту групу. Можна дати сильнішу
   модель/effort (`consolidate_model` / `consolidate_effort`) — це лише
   кілька викликів. Всі дії виводяться в лог; ручні overrides
   застосовуються після і мають пріоритет.
5. **Перепризначення** (`reassign`) — усі фрази ще раз розкладаються проти
   ГОТОВОЇ таксономії (нові групи заборонені). Ранні батчі першого пасу
   групувалися, коли таксономії ще не було, — цей пас розставляє все так,
   ніби повна структура була відома від початку. Голоси перераховуються з
   нуля (тож повторний прогін без `--fresh` більше не подвоює лічильники);
   рядки, що лишилися без голосів, і порожні групи видаляються. Точні
   збіги тексту з рядком розкладаються детерміновано без LLM.
6. **Черга перевірки** — сумнівні місця (розділені гейтом злиття, рядки,
   що розчинилися, дії консолідації, групи-одинаки, аварійні розміщення)
   пишуться у `review_queue.json` і показуються у вкладці «🕵 Перевірити»
   GUI — людина переглядає десятки місць замість всієї книги.
   Вердикти ✓/✗ зберігаються у `gate_labels.json` і **діють на наступних
   прогонах** (`pipeline/precedents.py`, налаштування — `gate_feedback` у
   config.yaml): точний прецедент вирішує ту саму пару автоматично; схожий
   прецедент (те саме порушене правило, лексично близька пара) успадковує
   вердикт; а правило гейта, чиї вето часто позначаються хибними, отримує
   «м'яку» вагу — злиття судді проходить, а місце пишеться у секцію
   «🧠 Зняті вето» черги для контролю (там теж є ✓/✗ — хибне зняття вето
   поверне фрази в окремі рядки з наступного прогону).
   **Усі три шари спільні для всіх товарів:** під час прогону мітки
   збираються з усіх `gate_labels.json` в один пул, тож ваш вердикт на
   одному товарі вирішує ту саму (точний прецедент) чи лексично дуже
   близьку (схожий прецедент) пару на будь-якому іншому, а ваги правил
   сумуються по всіх товарах. Запобіжники лишаються: перенесення лише при
   збігу категорії + високій лексичній схожості (для схожого — з обох
   боків), тож несхожі формулювання чужого товару не збігаються. Ваги
   правил додатково зберігаються у людиночитний файл
   `products/gate_rule_weights.json` (перебудовується при кожному прогоні
   й при збереженні мітки в GUI; видалити безпечно — це звіт).
7. **Excel** — 5 аркушів у форматі еталона, тотали, сортування за
   спаданням, кольорове маркування продуктів.

Всі LLM-виклики кешуються (`.llm_cache.json`) — перезапуск безкоштовний.
Таксономія зберігається у `taxonomy.json` — новий конкурент наповнює
існуючі групи, а не будує нові.

## Налаштування

```powershell
pip install -r requirements.txt
```

Впишіть ключ у файл `.env` у цій папці:

```
OPENROUTER_API_KEY=sk-or-...
```

(або `ANTHROPIC_API_KEY=...` якщо в `config.yaml` стоїть
`provider: anthropic`). Змінна середовища, якщо задана, має пріоритет над
`.env`. Параметри (провайдер, модель, effort, батчі, max_reviews,
double_check, consolidate) — у `config.yaml`. Для OpenRouter модель
задається слагом, напр. `anthropic/claude-sonnet-4.5`
(список: https://openrouter.ai/models?q=claude). `effort` вмикає extended
thinking для обох провайдерів — для групування це помітно додає точності.

## Графічний інтерфейс

```powershell
streamlit run app.py
```

Відкриється сторінка в браузері: завантаження PDF, редагування назв
продуктів, запуск пайплайна з прогрес-барами, перегляд таксономії з
пошуком і редактор корекцій (перейменування/злиття груп, перенесення
фраз) з валідацією та перегенерацією Excel без LLM-викликів.

## Запуск з консолі

```powershell
# 1. Покладіть PDF у папку продуктової лінійки, напр. products\styptic\
python run.py products\styptic          # перший запуск створить products.yaml

# 2. Заповніть у products.yaml короткі назви продуктів і посилання

# 3. Повний прогін
python run.py products\styptic

# Швидкий тест на 15 відгуках з кожного PDF:
python run.py products\styptic --limit 15

# Тільки відгуки після дати:
python run.py products\styptic --cutoff 2025-04-01

# Пропустити фінальний пас консолідації таксономії:
python run.py products\styptic --no-consolidate
```

Результат: `products\styptic\styptic - DD.MM.YYYY.xlsx`.

## Універсальність: різні групи товарів

Інструмент доменно-незалежний. Уся логіка (екстракція, дворівневе
групування, детермінований гейт злиття, консолідація) не містить лексики
конкретного товару, а промпти судді сформульовані загальними правилами
(дослівність, яруси похвали, кваліфікатори, негація, домінантний кут) без
прикладів із якоїсь однієї ніші. Єдиний виняток — `judge_examples` у
профілі домену (див. нижче): кілька *доменних* прикладів «як людина читає
неоднозначну фразу», які подаються судді як few-shot. Заміри на еталонному
файлі показали, що без них суддя стягує всі фрази зі згадкою головної
вигоди в одну мега-групу (F1 0.95 → 0.73), а абстрактні (позадоменні)
приклади не допомагають — тому приклади живуть у даних профілю, не в коді.

Що саме є доменним — набір **категорій**, семантика полів `relation`/`gist`
і розкладка **Excel-аркушів** — винесено у профіль `domain.yaml` у папці
продукту. Якщо файлу немає, діє вбудований профіль за замовчуванням
(п'ять категорій: positive / negative / usage / improvement /
who_recommended — як і раніше, поведінка не змінюється).

Для нової групи товарів створіть профіль-скелет і відредагуйте його:

```powershell
python run.py products\headphones --init-domain
```

`domain.yaml` описує категорії та аркуші:

```yaml
categories:
  - id: sound                 # id категорії (вживається скрізь у пайплайні)
    key: quote                # quote = дослівні рядки (діє гейт злиття);
                              # relation = короткі мітки-відношення (usage-стиль)
    extract_rule: "похвала або скарга на якість звуку."   # нейтральна вказівка екстрактору
    group_hint: "твердження про якість звуку."            # що таке ГРУПА в цій категорії
  - id: scenario
    key: relation
    subbucket: true           # relation: коарс-бакет (як usage_category) на аркуші
    extract_rule: "де/як використовують. Заповни relation сценарієм."
    group_hint: "сценарій використання."
  # name_from: gist            # (як improvement) — групу називають за міткою gist

sheets:
  - title: Sound
    layout: two_level         # two_level | relation | wish
    categories: [sound]
    headers: [Claim, "The reviews", Product, Votes, "By product", Total]
    widths: [30, 40, 16, 12, 12, 14]
  - title: Scenarios
    layout: relation
    categories: [scenario]     # один аркуш може збирати кілька relation-категорій
    headers: [Scenario, Relation, Product, "By product", "By relation", Overall]
    widths: [30, 34, 18, 16, 15, 12]
    bucket_labels: {scenario: Other}

# Доменні few-shot приклади для судді групування/перепризначення/консолідації:
# як людина розкладає фрази, що згадують головну вигоду мимохідь. Напишіть
# 2-4 свої (або видаліть ключ — діятимуть лише загальні правила).
judge_examples:
  - '"the bass is amazing even at low volume" — суть у якості звуку -> група
    про звук, а не про гучність, яку фраза згадує мимохідь.'
```

Три готові розкладки аркушів: `two_level` (USP/Negative — група → рядки →
голоси), `relation` (Usage-стиль — з коарс-бакетами), `wish`
(Improvement-стиль зі зразком відгуку). Аркуш «Products» додається завжди.
GUI підхоплює профіль автоматично (категорії, вкладки, назви аркушів).

## Ручні корекції

Створіть `overrides.json` у папці продукту:

```json
{
  "rename":           {"Multi-purpose": "Different sizes"},
  "rename_canonical": {"appears that it will work well": "works well"},
  "merge_groups":     [["Easy to use", "Simple to apply"]],
  "move_canonical":   {"very comfortable": "Great quality"}
}
```

`rename_canonical` перейменовує рядок (канонічну фразу) — наприклад, коли
рядок отримав назву за невдалим першим формулюванням.

і перегенеруйте Excel без повторних LLM-викликів:

```powershell
python run.py products\styptic --excel-only
```

Корекції застосовуються після кожного прогону, тож вони переживають
додавання нових продуктів.

## Калібрування судді

Перевірка групування на еталонній книзі (відтворює людські рішення і
показує розбіжності):

```powershell
python calibrate.py "..\Styptic powder - 25.05.2026.xlsx" --limit 150
```

Виводить попарні precision/recall/F1 щодо еталонних USP-груп, приклади
помилкових об'єднань/розділень і список злиттів на рівні фраз для ручної
перевірки.

## Тести

Юніт-тести детермінованого ядра (дедуплікація перекриттів, узгодження
голосів «один відгук = один голос», ідентичність продуктів, dual-placement,
гейт злиття, формули Excel). Зовнішніх залежностей не потребують:

```powershell
python -m unittest discover -s tests
```
