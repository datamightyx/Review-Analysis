# Review Scoring — версія 2: технічний опис

Документ для зовнішнього рецензента. Мета системи — автоматизувати SOP «рев'ю-скорінг»:
з PDF-експортів сторінок відгуків Amazon побудувати Excel-книгу, де дослівні фрази
покупців згруповані у теми (USP / негативи / способи використання / побажання),
з підрахунком голосів по продуктах і загалом. Еталон якості — книга, яку людина
робила вручну (Styptic powder — 25.05.2026.xlsx).

## 1. Загальна архітектура

Python 3.12, без БД — все у JSON-файлах у папці продукту. Два інтерфейси:
- `run.py` — CLI-пайплайн (основний шлях);
- `app.py` — Streamlit GUI (запуск, перегляд результатів, людська перевірка
  спірних місць, ручні корекції, облік витрат).

Конвеєр (усі кроки в `pipeline/`):

```
PDF-експорти відгуків Amazon
   │  pdf_parser.py  (regex-парсинг друкованої сторінки)
   ▼
reviews.json  (структуровані відгуки: автор, рейтинг, дата, варіація, текст)
   │  extract.py + llm.py  (LLM-екстракція дослівних цитат, батчами по 8)
   │  validate_verbatim()  (перевірка дослівності + ремонт + відкидання вигаданого)
   │  dedupe_overlapping() (усунення перекриттів фраз усередині одного відгуку)
   ▼
phrases.json  (цитата + категорія + продукт + review_id [+relation/gist])
   │  grouping.py  (інкрементальна побудова двохрівневої таксономії)
   │    4a. group_phrases      — LLM-суддя розкладає фрази батчами по 25
   │    4b. consolidate_taxonomy — LLM-пас по всій таксономії (злиття груп, перенос рядків)
   │    4c. reassign_phrases   — повторне розкладання ВСЬОГО корпусу у фінальну таксономію
   │    4d. merge_sibling_rows — LLM-пас злиття рядків-синонімів усередині груп
   ▼
taxonomy.json  (групи → канонічні рядки → голоси/цитати/review_ids по продуктах)
   │  apply_overrides()  (ручні корекції з overrides.json — переживають перезапуски)
   │  reconcile_votes()  (інваріант: 1 відгук = 1 голос)
   ▼
excel_writer.py → «<папка> - DD.MM.YYYY.xlsx» (Positive / Negative / Usage / Improvement / Products)
```

Побічні виходи: `verbatim_warnings.json` (вигадані цитати), `review_queue.json`
(місця для людської перевірки), `.llm_cache.json` (кеш LLM-викликів),
`gate_labels.json` (людські вердикти ✓/✗), `usage_history.jsonl` (витрати).

## 2. Модель даних (`models.py`)

Двохрівнева таксономія, дзеркалить еталонну книгу:

- **Group** — тема рівня 1 (один USP / один негатив / один тип використання).
  Поля: `id`, `category`, `name`, `usage_category` (грубий бакет для Usage-аркуша).
- **Canonical** — рядок рівня 2 усередині групи: одне «повідомлення», куди злиті
  майже-ідентичні формулювання. Поля: `text` (репрезентативна ДОСЛІВНА фраза),
  `votes {продукт: n}`, `quotes {продукт: [сирі варіанти]}`,
  `review_ids {продукт: [ідентифікатори відгуків]}`.
- **Taxonomy** — контейнер обох + лічильник id; серіалізується в `taxonomy.json`.

Ключові інваріанти:
1. **1 відгук = 1 голос у рядку.** `reconcile_votes()` перед КОЖНИМ записом
   перераховує `votes` з кількості унікальних `review_ids` — лікує будь-які
   подвійні підрахунки після злиттів.
2. **Голоси перебудовуються з корпусу на кожному прогоні.** Таксономія переносить
   між прогонами лише СТРУКТУРУ (групи + тексти рядків); голоси обнуляються і
   наповнюються заново — повторний прогін не подвоює рахунки.
3. **Dual placement** — одна цитата, що реально несе дві теми («durable and
   reusable»), свідомо рахується у двох групах/рядках; такі «близнюки»
   (однаковий normalized text у двох групах) захищені від автозлиття всюди.
4. `product_key()` — позитивний і негативний PDF одного продукту
   (`B0..._positive` / `B0..._negative`) згортаються в одну колонку продукту.

## 3. Парсинг PDF (`pdf_parser.py`)

PDF — це браузерні «друки» сторінок відгуків Amazon. Текст витягується
pdfplumber (fallback: PyPDF2), нормалізується до одного рядка, далі regex-розбір:
маркер відгуку — «N out of 5 stars»; сегмент між маркерами ріжеться на
автора / заголовок / дату / варіацію / тіло; хвіст «Helpful | Report» відсікає
відгук і дає автора наступного. Дедуплікація по (автор, дата, перші 120 символів),
сортування новіші-перші, опційні `--cutoff` і `--max-reviews` (реалізують правило
SOP «до 100 відгуків або до дати»).

Слабке місце за побудовою: якщо Amazon змінить верстку або PDF буде іншою мовою —
regex-и посипляться; парсер ніяк не валідує повноту (скільки відгуків «загубилось»).

## 4. Екстракція фраз (`extract.py`)

LLM-виклик на батч з 8 відгуків (`extract_batch`), окрема дешевша модель
(`extract_model`, зараз Haiku 4.5, effort low). Промпт кодує правила SOP:

- **дослівність** — цитата = один суцільний span з відгуку, без парафраз і склеювань;
- категорії генеруються з доменного профілю (див. §8), фраза може мати до двох;
- **розщеплення** перелічених аспектів («Not suitable for kids…, or facial cuts» →
  дві фрази) і заборона перекриттів у межах відгуку;
- **скіп безконтентного** («Perfect», «A+», доставка, ціна без value-складової);
- для usage заповнюється `relation` (об'єкт застосування), для improvement — `gist`.

Після екстракції два механізми контролю:

1. **`validate_verbatim()`** — кожна цитата перевіряється як substring
   (після normalize) свого відгуку. Не substring → пошук найкращого вікна
   (`difflib`, поріг 0.8) і «ремонт» до реального тексту; далі одна спроба
   LLM-ремонту (знайти справжній span); що не відремонтувалось — ВИДАЛЯЄТЬСЯ
   і логуються у `verbatim_warnings.json`. Гарантія: вигаданий текст не потрапляє
   в таксономію.
2. **`dedupe_overlapping()`** — детерміністично (без LLM): якщо у межах
   (продукт, відгук, категорія) фраза A є підсписком слів фрази B, то B або
   відкидається (перефраз того ж), або, якщо хвіст B — окрема сурядна клауза,
   розщеплюється (SPLIT): лишаються A і хвіст. Це захист від подвійного голосу
   одного відгуку.

## 5. Групування (`grouping.py`) — серце системи

### 5.1. Підготовка
Фрази групуються по категоріях; `_unique_phrases()` детерміністично зливає
точні normalized-дублікати (з підсумовуванням лічильників по продуктах) і сортує
за частотою. Для relation-категорій (usage, who_recommended) ключем є `relation`,
а не цитата.

### 5.2. Детерміністичний препас (`_deterministic_prepass`)
До будь-якого LLM-виклику: (а) точні normalized-збіги з наявними рядками лягають
одразу (включно з dual-placement близнюками); (б) `auto_merge_target()` —
детерміністичний «гейт» (див. §6) авто-зливає формулювання, які точно є тим самим
повідомленням («They work good» → «Works well»). Повторні прогони майже безплатні.

### 5.3. LLM-суддя (батчі по 25, `group_batch`)
Кожен батч бачить: поточну таксономію категорії (групи з топ-12 рядками, хвости
обрізані) + нові фрази з лічильниками і підказками `[similar: …]` — топ-5
лексично близьких рядків (token/trigram Jaccard, `similarity.py`) плюс до 3
семантичних сусідів (статичні ембедінги model2vec potion-base-8M,
`embeddings.py`; лише для recall кандидатів — рішення про злиття вони не
приймають, бо cosine анти-корелює з правилом SOP: «Works well»~«Works great»=0.87
НЕ зливати, «worked outstandin»~«Works great»=0.22 зливати).

Суддя для кожної фрази обирає: злити в наявний canonical / новий рядок у наявній
групі / нова група; підтримує посилання всередині батча (`same_group_as_phrase`,
`merge_with_phrase`), dual placement (`second_group_id`, `second_canonical_id`)
і `usage_category`. Промпт (~180 рядків) кодує пороги SOP — центральні правила:

- рівні похвали не змішуються: plain (good/well/works/does the trick) ≠ strong
  (great/excellent/perfect/fantastic…); всередині рівня — зливати;
- кваліфікатори — контент: «quickly» ≠ «instantly» ≠ «fast», таймфрейми
  розділяють; інтенсифікатори (very/really/so) — ні;
- заперечні скарги зливаються через форми негації і ступені («don't stick» ==
  «Didn't stick very well»), але інший об'єкт/аспект — окремо;
- довге багатодетальне речення ніколи не розчиняється в короткому рядку
  (деталі — контент-матеріал);
- ГРУПИ ШИРОКІ (тема = один булет, 10–25 груп на категорію), нюанс живе на
  рівні рядків; DOMINANT ANGLE — фразу класти за її головним смислом, а не за
  побіжно згаданою вигодою.

Пропуски судді не гублять голоси: `_fallback_place()` кладе фразу окремим рядком
у групу лексично найближчого рядка і флагує в `review_queue.json`.

### 5.4. Верифікація нових груп (`_verify_new_groups`, `double_check: true`)
Після кожного батча нові групи, лексично схожі (≥0.25) на наявні, переглядає
дешевший LLM-виклик: «дублікат теми → злити». Бореться з накопиченням груп-синонімів
під час інкрементальної побудови.

### 5.5. Консолідація (`consolidate_taxonomy`)
Один виклик на категорію по ВСІЙ таксономії (сильніша модель/effort — це кілька
викликів на прогін): пропонує (а) злиття груп-дублікатів, (б) переноси окремих
рядків у правильну групу (знову DOMINANT ANGLE). Застосовується одразу.

### 5.6. Перепризначення (`reassign_phrases`)
Головний прийом проти залежності від порядку батчів: ВЕСЬ корпус фраз ще раз
розкладається у ФІНАЛЬНУ (заморожену — нові групи заборонені) таксономію,
голоси перебудовуються з нуля; рядки без голосів і порожні групи зникають.
Таксономія категорії вшита в system-блок (prompt cache → ~0.1x ціни на
повторних батчах). Нижчий effort (задача — matching, не побудова).

### 5.7. Злиття рядків-синонімів (`merge_sibling_rows`, `row_merge: true`)
LLM-суддя по групах (пакування до 120 рядків на виклик): знаходить у межах групи
рядки з тим самим повідомленням різними словами («don't stick» == «won't hold» ==
«no adhesion») — те, чого лексичний гейт бачити не може. Найголосованіше
формулювання дає ім'я рядку. КОЖНА пропозиція проходить жорстке вето
`merge_blocked()` (§6); вето може зняти людський прецедент (§7).

## 6. Детерміністичний merge-гейт (`similarity.py`)

Дві функції з різною логікою за побудовою:

- **`merge_compatible(a,b)`** — «чи це ТОЧНО те саме повідомлення?» — строга,
  використовується для автозлиття БЕЗ судді. Пайплайн нормалізації: casefold →
  розкриття скорочень (n't → not) → мапінг ідіом («does the trick» → «works
  well», з умовою що ідіома — все повідомлення) → стоп-слова/інтенсифікатори →
  легкий стемінг. Далі: негація лише з одного боку — блок; різні множини
  кваліфікаторів — блок; обидві заперечні — порівняння після відкидання
  залишків негації/ступенів (+ опційні `synonym_families` з config.yaml);
  далі — логіка рівнів похвали + «works-патерн» (підмет/бенефіціар не блокує:
  «The incision worked fantastic» == «Works great»).
- **`merge_blocked(a,b)`** — «чи це ТОЧНО порушує поріг SOP?» — навмисно м'якша,
  вето лише на жорсткі правила (негація з одного боку, різні кваліфікатори,
  різні рівні похвали, довге-проти-короткого ≥4 стем-токени різниці). Все інше
  (семантична синонімія) — рішення судді. Причина асиметрії задокументована:
  блокування більшого дробило рядки на десятки майже-дублікатів.

## 7. Петля людського зворотного зв'язку (`precedents.py` + GUI-вкладка «🕵 Перевірити»)

Кожне вето гейта логовано (`gate_blocked` у `review_queue.json`); людина у GUI
ставить ✓ (вето правильне) або ✗ (треба було злити) → `gate_labels.json`.
На наступних прогонах вето розв'язується через три шари (сильніший перший):

1. **Точний прецедент** — та сама пара (category, phrase, into) вже розмічена:
   рішення людини відтворюється детерміністично (обидва напрямки пари).
2. **Близький прецедент** — розмічена пара з ТИМ ЖЕ правилом, обидві сторони
   лексично близькі (min по сторонах ≥0.75): вердикт переноситься.
3. **Ваги правил** — якщо ≥70% з ≥3 міток одного правила кажуть «зливати»,
   правило стає «м'яким»: вето знімається, місце логовано `gate_overridden`.

Усі три шари ГЛОБАЛЬНІ: `gate_labels.json` усіх папок продуктів зливаються в
один пул; агрегат ваг персистовано у `products/gate_rule_weights.json`.

## 8. Доменний профіль (`domain.py`) — універсальність

Все, що було зашито під wound-care домен, винесено в дані: список категорій
(з extract_rule / group_hint / key=quote|relation / name_from=gist / subbucket),
опис Excel-аркушів (SheetSpec: layout two_level|relation|wish, заголовки, ширини)
і `judge_examples` — доменні few-shots для суддів (виміряно: без них F1
калібрування падає 0.95 → 0.73, бо суддя стягує все, що згадує основну вигоду,
в одну мега-групу). Дефолтний профіль відтворює оригінальні п'ять категорій;
`python run.py <папка> --init-domain` генерує скелет `domain.yaml` під нову
товарну групу. Промпти екстракції та групування збираються з профілю — доменна
лексика в коді відсутня.

## 9. LLM-шар (`llm.py`)

- Два бекенди: Anthropic SDK і OpenRouter (зараз у config: OpenRouter,
  `anthropic/claude-sonnet-5`; екстракція — Haiku 4.5).
- Structured output: json_schema (strict) з fallback на «поверни JSON» + loose
  parsing; reasoning/effort параметр з автовідкиданням, якщо модель не підтримує.
- **Дисковий кеш** по SHA-256(provider, model, effort, system, user, schema) —
  перезапуск після падіння/твіку безплатний; merge-on-write (кілька інстансів
  ділять один файл).
- **Prompt cache** (cache_control ephemeral на system-блоці) — системні промпти
  однакові в межах кроку; у reassign таксономія свідомо перенесена в system.
- Ретраї з експоненційним бекофом на 429/5xx/transient-помилки в тілі 200.
- Облік токенів+вартості на кожен інстанс (окремі інстанси на екстракцію /
  групування / verify / reassign / консолідацію з окремими model/effort) +
  зведений звіт; GUI має вкладку «💰 Витрати» (usage_history.jsonl).

## 10. Excel (`excel_writer.py`)

openpyxl; лейаут керується доменним профілем. Три типи аркушів: two_level
(Positive/Negative: група → рядки → продукти, формули =SUM для Total by product /
Total by USP), relation (Usage: бакети usage_category смугами, спільний аркуш
usage + who_recommended), wish (Improvement: продукт-побажання-приклад-голоси).
Групи і рядки відсортовані за голосами спадно (вимога SOP). Форматування
скопійоване з еталонної книги (Arial, блакитна шапка, сіре бандування, freeze).
`save_workbook()` у run.py обходить PermissionError (файл відкритий в Excel)
суфіксами « (2)».

## 11. Калібрування (`calibrate.py`)

Метрика якості групування проти еталонної книги: бере рядки з gold-групами
(стратифікована вибірка по групах), проганяє ПОВНИЙ конвеєр групування
(group → consolidate → reassign → row_merge) і рахує попарні precision/recall/F1
відношення «в одній групі», плюс список підозрілих злиттів на рівні рядків
(еталонні рядки одного продукту вже дедупліковані людиною, тож їх злиття =
over-merge). Прецеденти (§7) у калібруванні НЕ встановлюються — міряється сирий
гейт. Окремий кеш `.calibrate_cache.json`.

## 12. GUI (`app.py`, Streamlit)

Сім вкладок: 📦 Продукти й PDF (завантаження, products.yaml) · ▶️ Запуск
(параметри + прогрес) · 🗂 Результати (таксономія по категоріях, графіки Altair)
· 🔎 Фрази товару · 🕵 Перевірити (review_queue: singleton-групи, fallback-и,
gate_blocked з ✓/✗-розміткою → gate_labels.json) · ✏️ Корекції (rename /
merge_groups / move_canonical / merge_canonicals / dual_place → overrides.json,
застосовуються після кожного прогону) · 💰 Витрати.

## 14. Відомі слабкі місця і рекомендації

### Якість результату (найімовірніші точки росту)
1. **Екстракція не вимірюється.** calibrate.py міряє лише групування; recall
   екстракції (скільки корисних фраз пропущено, скільки сміття взято) ніхто не
   рахує. Рекомендація: розмітити вручну 30–50 відгуків (gold-фрази) і додати
   `calibrate_extract.py` з precision/recall по спанах. Це дешево і, ймовірно,
   найбільший невидимий важіль якості — все, що загубила екстракція, не врятує
   жоден наступний пас.
2. **Категоризація на етапі екстракції жорстка.** Фраза, що отримала не ту
   категорію (positive vs usage vs improvement), уже не мігрує — групування
   працює всередині категорії. Рекомендація: пас перевірки категорій (дешевою
   моделлю) або дозволити консолідації пропонувати cross-category переноси.
3. **Порядок батчів усе ще впливає** (reassign прибирає більшість, але
   структура груп народжується в першому пасі). Рекомендація: A/B-прогін з
   перемішаним порядком на одному продукті та порівняння F1 — оцінити реальну
   величину проблеми, перш ніж щось міняти.
4. **Іменування груп** — «найчастіше формулювання покупця» інколи дає невдалу
   назву теми. Дешевий фінальний пас «перейменуй групи за їх вмістом
   (2–5 слів, найчастіше клієнтське формулювання)» з ручним ревю в GUI.
5. **Калібрування покриває один домен і один аркуш** (Positive/Negative
   стилю two_level). Usage/Improvement/who_recommended не міряються. Додати
   gold-зразки і для них.
6. **Ембедінги слабкі** (potion-base-8M, статичні). Оскільки вони лише
   постачають кандидатів, спробувати сильнішу модель recall-у (напр.
   bge-small через onnxruntime, без torch) і поміряти, чи зростає частка
   правильних злиттів «без спільних слів»; рішення все одно лишається за суддею.
7. **Праг \[similar:\]-хінтів** (top-5 лексика + top-3 семантика) — евристика;
   для великих таксономій судді показуються обрізані групи (top-12 рядків), і
   він може не побачити потрібний рядок. Варіант: динамічний контекст —
   показувати повністю лише групи, близькі до фраз батча.
